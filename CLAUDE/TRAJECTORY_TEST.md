# TrajectoryTest harness — reference (avoid re-scanning)

How `BallisticCalculator.Test/Calculator/TrajectoryCalculatorTest.cs` validates the engine
against reference trajectories. Verified against source 2026-07-13. Re-check cited file:line
if behavior surprises you.

## Files
- Test: `BallisticCalculator.Test/Calculator/TrajectoryCalculatorTest.cs`
- Loader: `BallisticCalculator.Test/Calculator/TableLoader.cs`
- Reference data: `BallisticCalculator.Test/resources/*.txt` (embedded resources,
  logical name `BallisticCalculator.Test.resources.<name>.txt`)
- **Reference provenance:** generated by an independent 3DOF model → treated as **ground truth**
  to converge toward (a fair external comparison, not our own regression output).

## Reference file format (`;`-separated, parsed by `TableLoader`)
Header rows (any order), then a units row, then numeric trajectory rows:
```
ammo;<BC>;<weight>;<muzzleVel>[;<diameter>;<length>]   # 6 fields => diameter+length (enables drift)
rifle;<sightHeight>;<zeroDist>[;<riflingStep>;<left|right>]  # 5 fields => rifling (enables twist)
wind;<velocity>;<direction>
atmosphere;<temperature>;<humidity%>;<pressure>;<altitude>   # humidity is PERCENT here, /100 in loader
shot;<shotAngle>;<cantAngle>
yd;in;moa;in;moa;ft/s;mach;ft-lb;s;in;moa                    # units row (first non-header, non-digit)
0;-1.5;0.0;0.0;0.0;3006.6;2.693;4415.0;0.000;0.0;0.0        # data rows
```
Data columns: `distance; drop; drop(moa); windage; windage(moa); velocity; mach; energy; time; ...`
(loader reads cols 0,1,3,5,6,7,8 → distance, drop, windage, velocity, mach, energy, time).
Values concatenate with the unit from the units row (e.g. `0` + `yd`).

## `TrajectoryTest` (the parameterized `[Theory]`, lines 63-109)
Signature: `TrajectoryTest(name, velocityAccuracyInPercent, dropAccuracyInMOA, windageAccuracyInMOA)`.
Runs `SightAngle` then `Calculate` with **Step=50 yd, MaximumDistance=1000 yd**, applies the
template's shot angle/cant and (single) wind, then asserts per point:
- row count equals template (`trajectory.Length.Should().Be(template.Trajectory.Count)`)
- **Distance** and **Velocity**: within `value * velocityAccuracyInPercent`
- **Drop** and **Windage**: within an inch tolerance derived from the MOA setting scaled by range
  (`MOA→InchPer100Yd × yards/100`), floored at 0.001 in.
- ⚠️ **`Time`, `Mach`, `Energy` are NOT asserted.** Fixes that only affect reported time
  (e.g. TOF truncation) are invisible here.

## Other tests in the file
- `Zero1`, `Zero_WithOffset` — `SightAngle` only, MOA tolerance 5e-2.
- `CustomTable` — same trajectory check via the `GC` custom-table path (rebinds BC to GC, passes
  a `DragTable` to both `SightAngle` and `Calculate`). vel 0.005 / drop 0.2 / wind 0.2.
- `CustomTableNotSpecifiedException` — GC BC without a table throws `ArgumentNullException`.
- `Custom1` — inline `MyDrag : DragTable` (32-pt custom curve), metric units, drop checked from i>0. vel 0.015 / drop 0.25.
- `Custom2` — `.drg` file via `DrgDragTable.Open`, 1500 m. vel 0.015 / drop 0.25.

## Coverage gaps (what the current resources do NOT exercise) — relevant to the a.* fixes
- **No file has twist AND shot angle together** → the spin-drift `×cos θ` fix (a2) is unverified here.
- **No steep shot** (max is 10° up) → steep-shot last-row termination (a4) barely/not triggered.
- **No Mach-increasing segment** → bidirectional drag-node walk (a5) not triggered.
- **Time not asserted** → TOF-accumulation fix (a3) not observable.
- Verifying a2/a4/a5 would require new reference files (twist+angle; steeper angle) sourced from
  the same external 3DOF model.

## Measuring margins (not just pass/fail)
Assertions only report pass/fail. To quantify improvement, run a diagnostic that computes per
file the max relative velocity error, max drop error (MOA), max windage error (MOA), and
row-count match — snapshot before/after a change, then tighten the `InlineData` tolerances to
the earned headroom.
