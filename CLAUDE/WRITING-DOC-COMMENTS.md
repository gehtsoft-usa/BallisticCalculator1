# Writing C# XML-doc comments that are compatible with docgen

The API docs for this library are generated by **Gehtsoft docgen** (the `cs2ds` template — see
`doc/prepareproject.xml`, built via the `doc/*.bat` scripts). docgen reads the compiler-emitted XML-doc
(`///` comments) but it is **not** MSDN/DocFX: it renders its own **bbcode** for inline markup and only
understands a subset of the standard XML-doc tags. Inline tags it doesn't understand (and raw angle
brackets) render wrong or break the layout.

Use this as the rule set whenever you add or edit `///` comments in `BallisticCalculator/`.
(Reference: commit `09cef14` "Fix docgen compatibility of b1 XML comments" is the worked example.)

---

## The rules (short version)

1. **First `<para>` is the brief, and it MUST be one physical line.** docgen takes the first paragraph
   of `<summary>` as the short description shown in lists/tables, and reads it as a **single line** —
   if the brief text wraps onto a second `///` line, the remainder is dropped / the brief renders
   broken. So keep the whole first `<para>…</para>` (open tag, text, close tag) on **one** `///` line,
   as one complete standalone sentence; put all detail in later `<para>` blocks (which may wrap freely).
   ```csharp
   /// <summary>
   /// <para>One-line brief sentence — the whole paragraph on a single line.</para>
   /// <para>Detail here may wrap across as many
   /// lines as you like.</para>
   /// </summary>
   ```
2. **Cross-references: `[clink=Namespace.Type]text[/clink]`, not `<see cref="…"/>`.** The target must be
   **fully qualified** (e.g. `BallisticCalculator.DrgDragTableFactory`). The visible text goes between
   the tags.
3. **Inline code: `[c]code[/c]`, not `<c>code</c>`.** Markdown backticks `` `code` `` also render, but
   `[c]…[/c]` is the established form for `///` comments here — prefer it.
4. **No angle brackets in prose — not even escaped.** Don't write `<`, `>`, `&lt;`, `&gt;`, or arrows
   like `A->B`. Write words: `Mach-to-BC` (not `Mach->BC`), "greater than", "less than or equal to".
   (Angle brackets only ever appear as the doc tags themselves.)
5. **Spell out mid-sentence abbreviations.** Use "for example" / "that is" instead of `e.g.` / `i.e.`
   — the trailing period confuses sentence splitting.
6. **Only these structural tags are safe:** `<summary>`, `<para>`, `<param name="…">`, `<returns>`,
   `<exception cref="…">`, `<value>`, `<remarks>`. Do **not** use inline `<see>`, `<c>`, `<code>`,
   `<list>`, `<paramref>`, `<typeparamref>` — convert them per rules 2–3 or reword.

That's it. Everything below is detail, examples, and the do/don't table.

---

## Do / don't

| Instead of (breaks / renders wrong)                     | Write (docgen-native)                                              |
|---------------------------------------------------------|-------------------------------------------------------------------|
| `<see cref="DrgDragTableFactory"/>`                     | `[clink=BallisticCalculator.DrgDragTableFactory]DrgDragTableFactory[/clink]` |
| `<see cref="DragTableId.GC"/>`                          | `[clink=BallisticCalculator.DragTableId]GC[/clink]` (or plain `GC`) |
| `<c>Cd_custom(M) = Cd_base(M) / BC(M)</c>`              | `[c]Cd_custom(M) = Cd_base(M) / BC(M)[/c]`                        |
| `a Mach-&gt;BC profile`                                 | `a Mach-to-BC profile`                                            |
| `values &gt; 0`                                         | `values greater than 0`                                           |
| `e.g. G1 or G7`                                         | `for example G1 or G7`                                            |
| single-line summary with no `<para>`                    | wrap the brief in the first `<para>` (see below)                  |
| brief `<para>` wrapped across two `///` lines           | keep the whole brief `<para>…</para>` on ONE line; wrap only later paragraphs |
| `<returns>A custom <see cref="DrgDragTable"/>.</returns>` | `<returns>A custom drag table.</returns>` (reword; no inline `<see>`) |

## Canonical example

```csharp
/// <summary>
/// <para>A single knot (a Mach number and the effective ballistic coefficient at it) of a multi-BC profile.</para>
/// <para>Consumed by [clink=BallisticCalculator.DrgDragTableFactory]DrgDragTableFactory[/clink] to synthesize a custom drag curve.</para>
/// </summary>
public readonly struct BcAtMach { … }

/// <summary>
/// <para>Synthesizes a custom drag table from a standard base drag curve and an effective ballistic-coefficient-vs-Mach profile.</para>
/// <para>The synthesized curve is [c]Cd_custom(M) = Cd_base(M) / BC(M)[/c], where [c]BC(M)[/c] is interpolated from the supplied knots.</para>
/// <para>The result is a custom [c]GC[/c] table; run it with a [clink=BallisticCalculator.BallisticCoefficient]BallisticCoefficient[/clink] value of 1.0 and table GC.</para>
/// </summary>
/// <param name="baseTable">The standard drag curve to scale (for example G1 or G7). Must not be GC.</param>
/// <param name="bcCurve">The Mach-to-effective-BC knots. Order does not matter and at least one knot is required. BC is interpolated linearly between knots and held flat beyond the end knots.</param>
/// <returns>A custom drag table on the base curve's Mach grid.</returns>
public static DrgDragTable Build(…) { … }
```

Notes:
- `<summary>` is split into `<para>` blocks; **the first is the brief**. Keep it self-contained.
- `<param>` / `<returns>` are plain prose — apply rules 2–5 inside them too (no `<see>`, no `>` / `<`).
- Reference another member's parameter by **name in prose** (`the specified Mach number`) rather than
  `<paramref>`.

## Notes & gotchas

- **Legacy files still contain `<see>`.** Many older files were written before these rules and still use
  `<see cref>` (docgen renders them as inert text, without a working link). Don't treat their presence as
  a green light — new and edited comments should use `[clink]`. Converting a legacy file opportunistically
  is welcome.
- **Verify after building docs**, not just after `dotnet build`. `dotnet build` only checks XML
  well-formedness (it will flag `CS1570`/`CS1587` for malformed tags, which is worth keeping at zero); it
  does **not** validate docgen bbcode. Run the docgen build (`doc/*.bat`, which needs the `%docgen%` tool
  on PATH; Windows) to see the rendered result.
- **Malformed-XML still matters.** Keep the comment valid XML (balanced tags, `&amp;` for a literal `&`).
  The `<` / `>` rule is about *prose*; the tag delimiters themselves are the only angle brackets allowed.

## Appendix — hand-written guide pages (`doc/src/*.ds`)

The task-oriented Guide is authored directly in docgen source (`.ds`) files, not from `///` comments, and
uses the same bbcode plus a few page-level constructs (reference: commit `d4e02f0`):

- **Tables:** a header row starts with `|!,` and cells are pipe-separated; body rows start with `|`:
  ```
  |!, Property | Meaning |
  | `Time`     | Time of flight to the point. |
  | `Drop`     | Vertical position relative to the line of sight. |
  ```
- **Internal links:** `[link=guide.output]Understanding the output[/link]` (anchor, not a type).
  Type links still use `[clink=Namespace.Type]text[/clink]`.
- **Inline code:** markdown backticks `` `Windage` `` render in `.ds` prose and tables.
- **Bulleted lists:** `* item` per line.

Keep example output blocks honest — the guide convention is to show real generated output (the docgen
`@show=yes` example directive), not hand-typed numbers.
